# Two Sum

## Варианты решения

### 1. Bruteforce — O(n²), O(1)
Двойной цикл. Перебираем все пары. Работает всегда, но медленно. Подходит только для n ≤ 1000.

### 2. Хеш-таблица — O(n), O(n)
Один проход. Для каждого числа смотрим, есть ли `target - num` в мапе уже просмотренных. Порядок важен: сначала проверяем мапу, потом кладём в неё — иначе дубликаты ломаются.

### 3. Сортировка + два указателя — O(n log n), O(n)
Сортируем массив, сохраняя исходные индексы. Ставим указатели слева и справа, двигаем к центру в зависимости от суммы.

## Что важно запомнить
- Классический компромисс «память за скорость»: хеш-таблица быстрее bruteforce ценой O(n) памяти

# Новые знания

## Quick Sort

Алгоритм «разделяй и властвуй». Выбираем опорный элемент (pivot), разбиваем массив на «меньше pivot» и «больше pivot», рекурсивно сортируем каждую часть.

**Сложность:**
- Средняя: O(n log n)
- Худшая: O(n²) — когда pivot всегда крайний (уже отсортированный массив)
- Память: O(log n) на стек вызовов (без учёта доп. массивов)

**Как работает partition:**
1. Берём pivot (обычно последний элемент)
2. Проходим по массиву, всё что меньше pivot — сдвигаем влево
3. В конце ставим pivot между «меньше» и «больше»
4. Возвращаем индекс pivot — массив разделён

### Схемы partition: Lomuto vs Hoare

| | Lomuto | Hoare |
|-------|--------|-------|
| pivot | Последний элемент | Первый элемент (или середина) |
| Указатели | Один указатель `i` слева | Два указателя `i` слева и `j` справа |
| Как работает | Всё что меньше pivot сдвигаем влево от `i`, в конце ставим pivot после `i` | `i` идёт вправо пока меньше pivot, `j` влево пока больше pivot, меняем местами |
| Возвращает | Индекс pivot (он на своём месте) | Точку раздела (`j`), pivot не обязательно на месте |
| Итераций | Один проход слева направо | Встречное движение, меньше сравнений |
| Сложность | Проще реализация, легче запомнить | Быстрее на практике, меньше свопов |

### Варианты выбора pivot

| Выбор | Плюс | Минус |
|-------|------|-------|
| Последний элемент | Просто | O(n²) на отсортированном массиве |
| Случайный | Защита от худшего случая | Доп. вызов rand |
| Медиана из трёх (первый, средний, последний) | Хороший компромисс | Чуть сложнее |

### Стабильность
Quick Sort **нестабилен** — одинаковые элементы могут поменяться местами при partition. Если нужна стабильность — Merge Sort.

### in-place vs с доп. памятью
- **in-place** (как здесь): меняем элементы в исходном слайсе, O(log n) на стек
- **с доп. слайсами**: создаём left и right слайсы, проще читается, но O(n) памяти
