# Longest Consecutive Sequence

## Инсайт

Сортируешь массив — очевидно O(n log n). O(n) без сортировки требует трюка: **считай последовательность только с её начала**.

Если `num-1` есть в сете — `num` не начало, его посчитают когда дойдут до `num-1`. Если `num-1` нет — это старт: иди вперёд пока `num+1` есть, считай длину.

Почему O(n)? Каждое число участвует не более чем в двух проверках: один раз как потенциальное начало, один раз как продолжение чьей-то последовательности во внутреннем цикле. Внутренний цикл не для каждого числа, а только для начал.

## Решение

### HashSet + обход начал — O(n), O(n)
1. Все числа в сет
2. Для каждого числа: если `num-1` нет — это начало → идём пока `num+1` в сете, считаем
3. Максимум длин

## Что важно запомнить
- Два этапа: заполнить сет, потом обработать. Не смешивать в одном цикле.
- Внутренний цикл идёт не по массиву, а проверяет `set[num+1]` — O(1) поиск
- `continue` без `else` — чище. После `continue` код и так не выполнится.
- Пустой массив → 0. Без этой проверки `res := 1` даст ложный ответ.
- Не надо хранить длины всех последовательностей — две переменные: текущая и максимальная.

## Что пошло не так
1. Сразу пытался считать частоты И обрабатывать в одном цикле — мой постойнный паттерн: смешивание этапов
2. Внутренний цикл писал по `nums[i:]` вместо проверки `set[num+1]` — O(n²) вместо O(n)
3. Синтаксис `for _, ok := set[num+1]; ok;` — ok вычисляется один раз, нужен `for { if ok { ... } else { break } }`
4. Забыл про пустой массив — `res=1` давал ложный ответ
5. TLE: модифицировал `num` во внутреннем цикле, а `num` — переменная `range`. Нужна отдельная `cur := num`.
6. TLE v2: итерация по `nums` (с дубликатами) вместо `set` (уникальные ключи). Одна и та же последовательность обрабатывалась повторно для каждого дубликата начала.
